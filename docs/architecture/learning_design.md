# Learning design

The application is not a passive digital textbook. It follows a repeatable cycle:

1. **Concise concept** — the minimum theory needed to reason correctly.
2. **Worked example** — a fully explained solution.
3. **Guided practice** — hints and progressively reduced support.
4. **Retrieval practice** — recall before seeing the answer.
5. **Immediate feedback** — why an answer is correct or incorrect.
6. **Transfer** — a new problem that requires adaptation.
7. **Spaced review** — later review based on performance.

## Supported activity families

- flashcards;
- single-answer and multiple-answer questions;
- true/false questions with justification;
- fill-in-the-blank and cloze activities;
- matching terms, concepts, outputs or pipeline stages;
- ordering algorithms, procedures and analytical pipelines;
- code completion, code tracing and debugging;
- short-answer and oral explanation prompts;
- interpretation of tables, plots and model outputs;
- pipeline design and integrative biomedical problems.

Each objective activity must include explanatory feedback. Open responses use rubrics and model answers rather than only a binary score.

## Optional Ollama integration

Ollama will be optional and configurable. The core application must remain fully usable offline without a language model.

Appropriate uses include:

- Socratic hints without immediately revealing the answer;
- alternative explanations of difficult concepts;
- oral-exam simulation;
- feedback on explanations or code;
- extra exercise variants generated from vetted templates.

The local model will not replace deterministic scoring, verified worked solutions, official course content or references. AI-generated output must be visibly identified.
